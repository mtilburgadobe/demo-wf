Wells Fargo's public website is a large-scale financial services site with approximately 1,050 indexed URLs organized into consumer banking, business banking, commercial/corporate banking, financial education, help/support, and corporate (about) sections. Roughly 42% of URLs are Spanish-language mirrors under /es/ that follow identical templates to their English counterparts.

Key Findings

  • 10 distinct page templates cover the entire site. Product landing pages (checking, savings, credit cards, mortgage) share a common template with hero + product cards + comparison table + FAQ pattern.
  • David's Model compliance is high — most content is structured text (headings, paragraphs, lists, links) that maps directly to EDS default content. Blocks are needed primarily for product cards, comparison tables, accordions, tabs, and promotional carousels.
  • 11 of 17 required blocks can be sourced from the repo or Block Collection (hero, cards, columns, accordion, tabs, carousel, table, video, embed, search, fragment). Six custom blocks are needed for Wells Fargo-specific patterns (product-comparison, rate-card, promo-banner, cta-strip, app-badge, contact-bar).
  • Spanish content uses identical templates and can be bulk-imported after English validation, requiring only i18n metadata adjustments.
  • Proprietary font "Wells Fargo Sans" must be licensed and self-hosted; fallback to system fonts for EDS.

Recommended Approach

Migrate template-by-template starting with the Homepage (highest visibility), then Product Landing Pages (largest template group), then Help/FAQ pages. Validate each template with a 5–10 page batch before scaling. Spanish mirrors import last as a bulk operation.

  • Language toggle (Español) will link to the /es/ mirror version of each page.

Import Workflow Per Template

  1. Analyze representative page → generate page-template + block parsers + transformers
  2. Import representative page → verify in local dev server
  3. Critique imported page against original → iterate CSS until visually acceptable
  4. Validate batch (5–10 pages) → identify parser edge cases → fix transformers
  5. Full import remaining pages → spot-check 10% sample

Spanish Mirror Strategy

Spanish pages use identical templates with translated content. Strategy:

  • Reuse the same block parsers and transformers as English
  • Add lang: es metadata to page-templates.json
  • Import after all English templates are validated
  • Spot-check 10 pages for character encoding / translation artifacts

8. Sequenced Worklist

Numbered phases in dependency order. Each phase builds on the previous.

Project Setup & Configuration

Configure fstab.yaml for DA, set up content folder structure, confirm EDS project is serving at *.aem.page. Add Wells Fargo Sans fonts to /fonts directory.

Dependencies: None (starting point)
Single-Page Migration — Homepage

Run site analysis → page analysis on wellsfargo.com/. Generate page template, block parsers, and transformers. Import homepage content. Verify all sections render in local dev server.

Dependencies: Phase 1 (project setup)
Design Phase 1 — Site-Level Styles

Extract and apply design tokens to styles/styles.css. Configure CSS custom properties for colors, typography, spacing. Set up responsive breakpoints (600px / 900px / 1200px).

Dependencies: Phase 2 (need imported content to style against)
Navigation Migration

Build nav.html with Brand / Sections / Tools structure. Build footer.html with legal sections using Fragment blocks. Customize header/footer block JS for mega-menu and disclaimers.

Dependencies: Phase 3 (site styles must be in place)
Block Collection Integration

Import Accordion, Tabs, Carousel, Table, Video, Embed, and Search blocks from Block Collection into the repo. Test each with sample content.

Dependencies: Phase 3 (need site styles for consistent look)
Custom Block Development

Build 6 custom blocks: Product-Comparison, Rate-Card, Promo-Banner, CTA-Strip, App-Badge, Contact-Bar. Create content models, JS decoration, and CSS for each.

Dependencies: Phase 5 (may compose with BC blocks)
Design Phase 2 — Block-Level Styles

Apply Wells Fargo visual treatment to all blocks. Match card styles, comparison table layout, accordion styling, carousel transitions to source site aesthetics.

Dependencies: Phases 5 & 6 (all blocks must exist)
Block Critique — Visual QA

Run block-by-block visual comparison against source site. Iterate CSS fixes until each block meets brand consistency threshold. Target ≥85% visual similarity.

Dependencies: Phase 7 (block styles must be applied)
Bulk Import — Validation Batches

Import 5–10 pages per template (9 templates × ~7 pages = ~63 pages). Validate each batch in local dev server. Fix parser/transformer edge cases discovered.

Dependencies: Phase 8 (blocks must be visually validated)
Page Critique — Full-Page QA

Run full-page visual comparison on validation batch pages. Identify cross-section layout issues, spacing inconsistencies, and missing content. Iterate fixes.

Dependencies: Phase 9 (validation pages must be imported)
Full Bulk Import — English Pages

Import all remaining English pages (~610 pages across all templates). Spot-check 10% sample (~61 pages) for quality.

Dependencies: Phase 10 (QA must pass on validation batches)
Full Bulk Import — Spanish Mirrors

Import ~438 Spanish pages using same parsers/transformers + lang: es metadata. Spot-check 10 pages for encoding and translation quality.

Dependencies: Phase 11 (English import must be complete and validated)
Project Documentation & Handover

Generate authoring guide, developer documentation, and admin guide. Document block content models, template usage, and publishing workflows for the Wells Fargo team.

Dependencies: Phase 12 (all content must be imported)

9. Risks & Mitigations

RiskSeverityLikelihoodImpactMitigation
Proprietary font licensing delays High Medium Blocks Design Phase 1 & 2 Start with system font fallback; swap fonts when license secured. Use font-display: swap.
Dynamic rate content cannot be statically authored High High Rate-Card block accuracy Author placeholder rates with "as of [date]" disclaimer. Explore client-side fetch from WF API.
Mega-menu complexity exceeds EDS header capabilities Medium Medium Navigation UX degradation Simplify to 2-level nav. Rich dropdowns via custom header JS. Prototype early in Phase 4.
Compliance/legal content requires exact reproduction High High Regulatory risk on disclaimer text Use Fragment blocks for legal text. QA with legal team. Automated diff checks on disclaimer content.
Spanish mirror pages have structural differences from English Medium Low Bulk import failures for /es/ pages Validate 10-page batch first. Build separate transformer rules for Spanish-only structural patterns.
Product comparison tables vary in column count per page Medium Medium Product-Comparison block breaks on some pages Build block to handle 2–4 products dynamically. Test with validation batch across product types.
External link breakage (creditcards.wellsfargo.com, etc.) Low Low Broken CTAs on product pages Catalog all external links during import. Implement link-checker in validation phase.
Import parser fails on edge-case page layouts Medium High 5–10% of pages need manual fixes Expected for large sites. Budget 2–3 parser iterations per template. Flag pages for manual review.
Performance regression from custom blocks Medium Low LCP / CLS degradation Follow EDS performance best practices. Run PageSpeed Insights on each template. Lazy-load below-fold blocks.
Content freshness — pages change between analysis and import Low Medium Stale content on initial import Minimize time between analysis and import per template. Re-import changed pages at cutover.

10. Open Questions

  1. Font licensing: Do we have rights to self-host "Wells Fargo Sans" web fonts in the EDS project? If not, what is the approved fallback?
  2. Rate data strategy: Should the Rate-Card block display static authored rates, or should we implement a client-side API fetch? Is there a public rates API available?
  3. Mega-menu scope: Should we aim to replicate the full mega-menu experience with product icons and descriptions, or simplify to a standard multi-level dropdown?
  4. Spanish content parity: Are all English pages guaranteed to have Spanish mirrors, or are some /es/ pages unique? Should we import all /es/ pages or a curated subset?
  5. Legal review process: Who reviews the migrated disclaimer/compliance content for accuracy? Is there a legal sign-off gate before go-live?
  6. Personalization replacement: The current site shows location-based content (ZIP code targeting). Is there a replacement strategy for EDS, or do we author generic national content?
  7. Analytics requirements: Which MarTech tags need to be re-implemented in delayed.js? Adobe Analytics? Google Analytics? Bing? All?
  8. Calculator tools: Are the financial calculators (affordability, DTI, budget) being rebuilt separately, or should we plan iframe embeds on EDS pages?
  9. Content freeze window: Can we coordinate a content freeze on the source site during the final bulk import to prevent drift?
  10. Go-live strategy: Will this be a big-bang cutover or a phased rollout by section (e.g., /about/ first, then /checking/, etc.)?
  11. Redirect mapping: Do any URLs change between the current site and EDS? If so, who maintains the redirect map?
  12. Accessibility audit: Is a WCAG 2.1 AA audit required before go-live? If so, who performs it?
